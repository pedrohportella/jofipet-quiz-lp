# Story 4.4: Lighthouse CI + quality gates no PR

## Status

Draft

## Story

**As a** dev,
**I want** PR bloqueado automaticamente se performance ou a11y regredir,
**so that** qualidade não degrada silenciosamente entre iterações.

## Acceptance Criteria

1. GitHub Action `.github/workflows/lighthouse.yml` executa Lighthouse CI em preview Vercel de cada PR
2. Barras: Performance mobile ≥85, Accessibility ≥95, Best Practices ≥90, SEO ≥90
3. Falha em qualquer barra bloqueia merge (required status check)
4. Testes Vitest também obrigatórios no PR com coverage mínimo 70% para arquivos em `src/lib/`
5. README documenta como rodar Lighthouse local + como ajustar barras

## Dev Notes

**Source artifacts:**
- Architecture §15 (deploy strategy + LH CI), §9 (testing requirements)
- PRD NFR1, NFR2 (performance), NFR4 (a11y)

**Dependency:** Stories 1.1 (workflow placeholder criado), 2.1 (hero existe para testar), 2.2 (captura), 3.1 (resultado).

**Critical rules:**
- Branch protection é @devops exclusivo (rule agent-authority)
- CI não deve rodar em commits marcados com `[skip ci]` (escape hatch)

**Barras justificativa:**
- Perf 85: PRD NFR1 exige LCP <2s — 85 no LH mobile é piso consistente
- A11y 95: WCAG AA é meta — 95 dá margem para edge cases (labels dinâmicas)
- Best Practices 90: standard
- SEO 90: LP pública merece

**LHCI config server:** Opcional usar LHCI server público ou `--upload.target=temporary-public-storage` para comentário no PR sem infraestrutura.
